# Insights — Change Data Capture (CDC) System

## Insight 1: CDC Solves the Dual-Write Problem by Eliminating It — Not by Making Dual Writes Safer, but by Making the Database's Own Transaction Log the Single Source of Truth for Both State and Events

**Category:** Architecture

**One-liner:** The fundamental insight of CDC is that every database already produces a perfect, ordered, complete stream of change events — its transaction log — and the correct architecture is to tap that existing stream rather than generating a second, unreliable stream from application code.

**Why it matters:** In a microservices architecture, a service that updates its database AND publishes a message to a broker is performing a dual write: two separate systems that cannot be updated atomically without distributed transactions. If the database write succeeds but the message publish fails (network error, broker down), the database and the event stream diverge permanently. If the publish succeeds but the database write fails (constraint violation, timeout), downstream consumers act on an event that never actually happened. The classic "solutions" — retry with idempotency, outbox polling, two-phase commit — all add complexity and have edge cases. CDC eliminates the problem entirely by recognizing that the database's transaction log IS the event stream. Every committed write is already recorded in the WAL/binlog with perfect ordering, atomicity, and completeness. CDC simply reads this existing stream and publishes it to a streaming platform. The outbox pattern leverages this: the application writes domain events to an outbox table in the same database transaction as the business data. CDC captures the outbox table and publishes the events — no dual write needed because the outbox write and the business data write are in the same transaction. The architectural insight is that you should never try to synchronize two writes; instead, derive the second from the first.

## Insight 2: The Snapshot-to-Streaming Handoff Is the Defining Engineering Challenge of CDC — Because It Requires Merging Two Fundamentally Different Data Sources (Full Table Scan and Transaction Log) into a Single Consistent Event Stream with Zero Gaps and Zero Duplicates

**Category:** Data Consistency

**One-liner:** The transition from initial snapshot (SELECT * FROM table) to live streaming (WAL tailing) must produce an event stream that is indistinguishable from one that captured every change from the beginning of time — and achieving this requires precise coordination of database snapshot isolation, log sequence number recording, and deduplication logic at the boundary.

**Why it matters:** When a CDC connector starts for the first time, it needs to capture the full current state of each table (the snapshot) and then switch to streaming live changes from the transaction log. The danger is at the boundary: changes that occur during the snapshot might be captured by both the snapshot SELECT (if the row was read before the change) and the streaming log (because the WAL entry was written during the snapshot window). Or worse, a change might be captured by neither — the snapshot already passed that row, and the streaming starts from an LSN after the change. The standard approach uses REPEATABLE READ isolation: the snapshot transaction sees a consistent point-in-time view, and the connector records the LSN at the start of the snapshot. After the snapshot completes, streaming begins from that recorded LSN. Events from the WAL between the snapshot's LSN and the current position are compared against the snapshot — if the snapshot already captured the newer state, the WAL event is redundant and skipped. Netflix's DBLog framework introduced a more sophisticated approach using watermarks: instead of a single long-running snapshot transaction (which can block vacuum on PostgreSQL and cause table bloat), DBLog processes tables in chunks, writing watermark tokens to a signal table between chunks. The watermarks appear in the WAL stream, allowing the connector to precisely interleave snapshot data with live changes on a per-chunk basis, without long-running transactions and without locks. Understanding both approaches — and their trade-offs (simplicity vs. source impact) — is what separates a textbook CDC answer from a production-ready one.

## Insight 3: A Stalled CDC Connector Can Take Down the Source Database — Because the Replication Slot Prevents WAL Recycling, and Unbounded WAL Growth Fills the Disk, at Which Point the Database Stops Accepting All Writes

**Category:** Operational Risk

**One-liner:** The most dangerous failure mode in a CDC system is not data loss or event duplication — it is the feedback loop where a stalled connector causes unbounded WAL growth on the source database, leading to disk exhaustion and a complete database outage that affects every application, not just the CDC pipeline.

**Why it matters:** On PostgreSQL, a replication slot guarantees that WAL segments are retained until the consumer (CDC connector) confirms it has processed them. This is essential for exactly-once delivery — without it, the database might recycle WAL segments before the connector reads them, causing permanent data loss. But this guarantee has a lethal flip side: if the connector stops consuming (crash, downstream outage, network partition, misconfiguration), the replication slot still prevents WAL recycling. On a database doing 100 MB/sec of writes, a 1-hour connector outage accumulates 360 GB of WAL. A 24-hour outage accumulates 8.6 TB. When the disk fills up, PostgreSQL enters a panic state — it cannot write new WAL entries, which means it cannot commit any transaction, which means every application using this database is completely blocked. The CDC connector's failure has cascaded into a full database outage. The operational implications are critical: (1) set `max_slot_wal_keep_size` to cap WAL retention per slot (PostgreSQL 13+), accepting that exceeding this limit drops WAL entries and requires re-snapshot; (2) monitor replication slot lag as the highest-priority CDC metric, with alerts at thresholds well before disk capacity; (3) implement heartbeat events — periodic writes to a heartbeat table — that advance the replication slot even when monitored tables have no activity. MySQL handles this differently: binlogs expire after a configured retention period regardless of consumers, which means a stalled connector loses events rather than threatening the database — a different failure mode (data loss vs. disk exhaustion) with different operational responses.

## Insight 4: Schema Evolution in CDC Is a Distributed Versioning Problem — Because the Same Physical Column Change Must Be Interpreted Correctly by Every Consumer That Has Ever Registered a Different Version of the Schema

**Category:** Data Consistency

**One-liner:** A single ALTER TABLE in the source database creates a schema versioning cascade: the CDC connector must detect the DDL from the WAL, register a new schema version with compatibility validation, and emit events that allow both old and new consumers to deserialize correctly — and the schema history store (keyed by LSN) is the mechanism that makes this possible.

**Why it matters:** In a traditional monolithic application, a schema change is a deployment event — you update the code and the schema together. In a CDC-driven architecture, the schema change originates at the source database, propagates through the connector, gets registered in the schema registry, and is then consumed by N independent downstream services — each potentially running a different version of their consumer code. The schema registry enforces compatibility rules (backward, forward, full) to ensure that this distributed versioning does not break consumers. But the subtlety is in the WAL timeline: the connector is processing a stream of WAL entries, and at some LSN position, the schema changes. Events before that LSN must be serialized with the old schema; events after must use the new schema. The schema history store — an internal mapping of LSN → schema — ensures this boundary is respected. Without it, the connector might serialize a pre-DDL event with the post-DDL schema, corrupting fields or losing data. The deeper architectural lesson is that in event-driven systems, schema is not a static contract but a time-varying function, and any system that streams data over time must track schema as a first-class versioned entity indexed by position in the event stream.

## Insight 5: The Outbox Pattern Transforms CDC from Infrastructure into Application Architecture — Because It Gives Applications Control Over the Shape, Timing, and Content of Domain Events Without Sacrificing Transactional Guarantees

**Category:** Design Pattern

**One-liner:** By writing domain events to an outbox table in the same transaction as the business data, applications decouple event design from physical schema design — CDC captures the outbox and publishes events that are shaped for consumers, not shaped by how the database stores data internally.

**Why it matters:** Raw CDC events expose the physical database schema to consumers: table names, column names, data types, and internal relationships all leak into the event stream. This creates tight coupling — renaming a column breaks every consumer. The outbox pattern solves this by introducing an explicit contract layer. The application writes a domain event (e.g., `OrderPlaced` with curated fields) to an outbox table within the same database transaction that modifies the business tables. CDC captures the outbox table and publishes those domain events. Consumers see a stable, designed API — not a raw database diff. The transactional guarantee is preserved because the outbox write and the business write are in the same ACID transaction — if the transaction rolls back, the outbox entry never existed, so no spurious event is ever published. The pattern also enables event enrichment: the application can compute derived fields, flatten joins, or include contextual information that doesn't exist in any single table. The trade-off is that applications must now explicitly produce events (additional code), and the outbox table itself becomes a write amplification point. But for any CDC system where consumers need domain semantics rather than physical schema diffs, the outbox pattern is the standard architectural choice.

## Insight 6: Exactly-Once Delivery in CDC Is Not a Property of Any Single Component — It Is an End-to-End Rule that never changes That Requires Coordinated Guarantees Across the Producer, the Streaming Platform, and Every Consumer

**Category:** Distributed Systems

**One-liner:** Claiming "exactly-once" by pointing to a single component (idempotent producer, transactional writes) is architecturally incorrect — true exactly-once is a pipeline-wide property that requires idempotent producers, transactional offset commits, and idempotent consumers all working together, and any break in this chain degrades to at-least-once.

**Why it matters:** The streaming platform's "exactly-once" feature (transactional writes with atomic offset commits) eliminates duplicates within the platform itself: a message is written to a topic partition exactly once, and the connector's offset advances atomically. But this guarantee stops at the platform boundary. If a consumer reads the event, applies it to a search index, and crashes before committing its consumer offset, it will re-read and re-apply the same event after restart — resulting in duplicate processing. For exactly-once to hold end-to-end, the consumer must also be idempotent: it must use the event's primary key and offset as a deduplication key, writing the data and the consumer offset in a single atomic operation (or using idempotent writes where re-applying the same event is a no-op). This three-layer guarantee — idempotent producer (dedup at publish), transactional platform write (dedup at storage), idempotent consumer (dedup at apply) — is the only architecture that delivers true exactly-once from source to sink. The practical implication is significant: every consumer must be designed for idempotency from the start. This is trivial for upserts (key-value stores, search indexes) but complex for operations that trigger side effects (sending emails, calling external APIs). For side-effect-producing consumers, the recommended pattern is to separate the event processing (idempotent) from the side effect (triggered by a separate, deduplication-aware component).

## Insight 7: CDC Connectors Are Logical Replicas of the Source Database — and Understanding Database Replication Internals (WAL Format, Logical Decoding, Replication Protocols) Is the Foundation for Understanding CDC Architecture

**Category:** Mental Model

**One-liner:** A CDC connector is architecturally identical to a database read replica, except instead of applying WAL entries to a local data directory, it transforms them into events and writes them to a streaming platform — and this mental model unlocks the correct thinking about consistency, failover, lag, and recovery.

**Why it matters:** Every concept in CDC maps directly to a database replication concept: the replication slot is the subscription, the LSN is the position in the replicated stream, connector lag is replication lag, connector failover is replica promotion, and the snapshot is the initial base backup. This mental model is powerful because database replication is a solved problem with decades of engineering behind it, and the solutions (slot management, WAL archiving, GTID-based failover, logical decoding) transfer directly to CDC. For example, PostgreSQL's logical replication uses the same infrastructure as CDC: `pg_logical` slots, the `pgoutput` plugin, and the streaming replication protocol. The connector connects to the database as a replication client (not a regular SQL client), receives WAL entries via the replication protocol, and acknowledges them to advance the slot. Understanding this means that WAL-level concepts like `restart_lsn` vs. `confirmed_flush_lsn` (the restart position vs. the acknowledged position) become intuitive rather than opaque. The failover behavior also becomes clear: when the primary fails and a replica is promoted, the CDC connector needs to connect to the new primary and find its position in the new primary's WAL — which is exactly the same problem that any replica faces during failover. GTID-based positioning (MySQL) or failover slots (PostgreSQL 17+) solve this for both replicas and CDC connectors identically.

## Insight 8: The Heartbeat Table Is a Deceptively Simple Mechanism That Solves Three Distinct Problems — Idle Slot Advancement, Lag Monitoring Accuracy, and Connector Liveness Detection

**Category:** Operational Pattern

**One-liner:** Periodically inserting a row into a dedicated heartbeat table ensures that (1) the replication slot advances even when monitored tables have no activity, preventing WAL accumulation from idle databases, (2) lag measurements reflect actual connector processing delay rather than "time since last event," and (3) the absence of heartbeat events at the consumer reliably signals connector failure.

**Why it matters:** Without heartbeat events, a CDC connector monitoring a database that goes idle (no writes to captured tables for hours) will appear stalled: its replication slot stops advancing, WAL accumulates from system catalog changes and other non-captured activity, and the lag metric becomes meaningless (it shows "0 events behind" because there are no events, but the slot is actually retaining gigabytes of WAL). The heartbeat table solves all three problems with a single mechanism: a periodic INSERT (every 10 seconds) into a `cdc_heartbeat` table. Because the heartbeat write goes through the WAL like any other transaction, it forces the replication slot to advance — preventing WAL accumulation from idle periods. Because the heartbeat event has a source timestamp, the consumer can compute the true end-to-end lag by comparing `heartbeat.source_ts` with `now()` — this gives a real-time measure of how far behind the pipeline is, even when business tables are idle. And because heartbeats arrive at a known frequency, their absence is a reliable signal: if no heartbeat arrives for 3× the heartbeat interval, the connector is likely dead or stalled. This three-in-one utility from a trivial mechanism is a hallmark of well-designed operational infrastructure — the best solutions are often the simplest ones, deployed at the right layer.

## Insight 9: Large Transactions Are the CDC Equivalent of Elephant Flows in Networking — They Monopolize Pipeline Resources, Delay All Subsequent Events, and Require Special Handling That Differs Fundamentally from Normal Event Processing

**Category:** Performance

**One-liner:** A single transaction that modifies 10 million rows produces 10 million WAL entries that must be buffered, deserialized, and published before the connector can advance to the next transaction — and this "head-of-line blocking" effect can stall the entire CDC pipeline for minutes unless the connector implements streaming transaction support or event-level chunking.

**Why it matters:** In log-based CDC, the connector reads the WAL sequentially. When it encounters a large transaction (batch UPDATE, bulk INSERT, partition maintenance), it must process all events within that transaction before advancing the LSN. If the connector buffers the entire transaction in memory before publishing (to preserve transactional atomicity in the event stream), a 10-million-row transaction might require gigabytes of memory — exceeding the connector's heap and causing an out-of-memory crash. If the connector publishes events as they arrive (streaming mode), memory is bounded, but consumers see a partial transaction that is eventually completed by a "transaction committed" marker event. The trade-off is clear: buffered mode preserves transactional boundaries but has unbounded memory requirements; streaming mode has bounded memory but pushes transaction awareness to consumers. Modern CDC connectors (Debezium 2.x+) implement streaming transactions: they emit events as they are read from the WAL, tagging each event with a transaction ID and emitting explicit BEGIN and END markers. Consumers that need transactional boundaries must buffer on their side — but this pushes the memory burden to the consumer, which can make consumer-specific decisions about how to handle it. The architectural lesson is that transaction boundaries are a source-side concept that doesn't map cleanly to streaming pipelines, and the connector must decide where the buffering responsibility lies.

## Insight 10: CDC Event Ordering Guarantees Are Per-Partition, Not Global — and the Partitioning Strategy Determines Which Consistency Properties Consumers Can Rely On

**Category:** Consistency Model

**One-liner:** Events for the same primary key are always ordered (same partition), but events across different keys in the same table may be reordered, and events across different tables have no ordering guarantee — which means consumers that need cross-key or cross-table consistency must implement their own ordering logic, not assume the streaming platform provides it.

**Why it matters:** CDC events are partitioned by primary key (hash of the key determines the partition). This guarantees that all events for a single row (INSERT, UPDATE, UPDATE, DELETE) arrive at the consumer in the same order they were committed in the source database. But two different rows' events may land on different partitions and be consumed at different rates. A consumer that needs to maintain a foreign key relationship (e.g., an order's `customer_id` references a customer row) cannot assume that the customer INSERT arrives before the order INSERT — even if they were committed in that order at the source. This is not a bug; it is a fundamental property of partitioned streaming. The solutions are consumer-side: (1) lookup joins — the consumer queries the target store for the referenced entity and retries if not found, (2) delayed processing — buffer events and process them after a configurable delay to allow referenced events to arrive, (3) transaction metadata — use CDC transaction markers to group events from the same source transaction and process them together. For strict cross-table ordering within a transaction, the outbox pattern provides a single-table solution that sidesteps the partitioning problem entirely. The key insight is that CDC provides per-key ordering and cross-table eventual consistency — and the consumer's consistency requirements should drive the consumption strategy, not the other way around.

## Insight 11: Incremental Snapshots (Watermark-Based) Represent a Fundamental Advance Over Traditional Snapshots — Because They Eliminate the Need for Long-Running Transactions and Global Locks by Interleaving Snapshot Chunks with Live Streaming Using Signal Table Watermarks

**Category:** Innovation

**One-liner:** Netflix's DBLog approach of writing low/high watermarks to a signal table, letting them appear in the WAL alongside live changes, and using them to precisely determine which snapshot chunks are consistent with which WAL positions — eliminates the core weakness of traditional snapshots (long-running transactions that block vacuum and cause table bloat) while maintaining full consistency.

**Why it matters:** Traditional CDC snapshots open a long-running REPEATABLE READ transaction to get a consistent view of the table, then read it in chunks. On PostgreSQL, this has a severe side effect: the snapshot transaction prevents the autovacuum from cleaning dead tuples in the table, because the transaction's snapshot might still reference them. For a billion-row table that takes 4 hours to snapshot, that is 4 hours of dead tuple accumulation — potentially causing table bloat, index bloat, and degraded query performance on the source database. The watermark-based approach eliminates this: instead of a single long transaction, the connector snapshots one chunk at a time (e.g., 10,000 rows), writing a "low watermark" to a signal table before the chunk and a "high watermark" after. These watermarks appear in the WAL stream, creating a precise bracket around each chunk. Events from the WAL that fall within a chunk's bracket are resolved against the snapshot data using a deterministic rule: if the WAL event's key was included in the snapshot chunk, the snapshot version takes precedence; if not, the WAL event is passed through. Between chunks, the connector processes live WAL events normally. This interleaving means no long-running transaction, no vacuum blocking, and the ability to pause/resume snapshots or snapshot individual tables on demand. The trade-off is implementation complexity: the watermark resolution logic must handle edge cases (what if a row is modified between the low watermark write and the chunk read? what if the chunk boundary splits a range that was modified by a single transaction?). But the operational benefits — no source impact, resumability, per-table granularity — make watermark-based snapshots the state of the art for production CDC systems.

## Insight 12: CDC Pipelines Create an Invisible Dependency Graph Across Your Organization — Where a Schema Change by One Team Can Silently Break Event Consumers Maintained by Ten Other Teams, Making Cross-Team Governance and Schema Registry Compatibility Rules Not Optional Best Practices but Essential Infrastructure

**Category:** Organizational

**One-liner:** Without centralized schema governance, CDC becomes a distributed coupling mechanism where every ALTER TABLE is a potential incident for every downstream consumer — and the schema registry's compatibility modes (backward, forward, full) are the only automated guard against this invisible dependency graph breaking at 3 AM.

**Why it matters:** In a microservices organization, the team that owns the source database may not know — or care — which other teams consume its CDC events. A developer adding a column (backward compatible) or renaming a column (backward incompatible) is making a local decision with distributed consequences. Without guardrails, this change flows through CDC to the streaming platform, where consumer A (search indexer) handles it gracefully (ignores unknown fields), consumer B (analytics pipeline) crashes because its Avro reader expects the old schema, and consumer C (cache updater) silently drops the data because the renamed field doesn't match its mapping. The schema registry's compatibility enforcement is the circuit breaker for this failure mode: if the registry is configured for backward compatibility, the connector's attempt to register a schema that removes a required field is rejected — the connector halts, an alert fires, and the breaking change is caught before it reaches any consumer. This transforms schema evolution from a "hope consumers handle it" approach to a "guarantee consumers can handle it" contract. The organizational implication is that the CDC platform team must mandate compatibility modes per topic and provide tooling for schema change previews (e.g., "will this ALTER TABLE pass the compatibility check?"). Without this governance, CDC's greatest strength — decoupled real-time data flow — becomes its greatest liability: a coupling mechanism that is invisible until it breaks.
