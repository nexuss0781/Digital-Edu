# Insights — NewSQL Database

## Insight 1: The Read Uncertainty Interval Is the Price You Pay for Commodity Clocks — and Its Width Directly Determines Your Transaction Restart Rate, Making Clock Quality a Performance Variable, Not Just an Infrastructure Detail

**Category:** Clock Synchronization

**One-liner:** Without atomic clocks, every read in a distributed database carries an uncertainty window during which it cannot distinguish whether a concurrent write happened before or after the read started, and the width of this window — set by NTP quality — controls how often transactions must restart to preserve serializability.

**Why it matters:** In a NewSQL database using hybrid logical clocks, each transaction reads at a timestamp `T` but cannot be certain that the real time at other nodes is not as far ahead as `T + max_clock_offset`. When a reader encounters a committed value with a timestamp between `T` and `T + max_clock_offset`, it faces an unresolvable ambiguity: the value might have been committed before the read started (should be visible) or after (should not be visible). The only safe resolution is to restart the transaction at a higher timestamp, which adds latency and wastes work. With default NTP (250ms offset), roughly 2-5% of transactions on a write-heavy workload experience at least one restart. With well-tuned NTP using chrony (10-50ms offset), the restart rate drops below 0.1%. With PTP (Precision Time Protocol, <1ms offset), restarts become negligible. This reveals a non-obvious architectural truth: clock synchronization quality is not an infrastructure checkbox but a first-class performance tuning parameter, as impactful as index design or cache sizing. Systems with atomic clocks (like Spanner) solve this by waiting out the uncertainty interval (~7ms), converting a probabilistic problem into a deterministic latency cost. Systems without atomic clocks convert it into a probabilistic restart cost. The engineering team must choose which cost they prefer — and most teams underestimate the restart cost until they observe it in production under contention.

## Insight 2: Parallel Commits Transform the Distributed Transaction Slowest part of the process from Two Sequential Consensus Rounds to One — but the Price Is That Any Node in the System Must Be Able to Determine a Transaction's Fate by Inspecting Its Intents

**Category:** Distributed Transactions

**One-liner:** Traditional two-phase commit requires two sequential rounds of Raft consensus (prepare, then commit), but parallel commits collapses this to one round by introducing a STAGING state where the transaction's commit status is implicitly determined by whether all its write intents are durable — shifting the recovery burden to any future reader that encounters an ambiguous intent.

**Why it matters:** In a traditional 2PC protocol on a consensus-based system, a transaction that writes to ranges R1 and R2 must first achieve consensus on both intents (round 1), then achieve consensus on the commit record (round 2). Each round adds the Raft consensus latency (~5ms local, ~100ms cross-region), making the total commit latency 2x consensus — the dominant cost for distributed transactions. Parallel commits eliminates the sequential dependency: the coordinator writes the STAGING transaction record concurrently with the final intent writes, and the moment both the STAGING record and all intents are durable, the transaction is committed. The client receives confirmation in one consensus round-trip. But this optimization creates a new responsibility: any node that later encounters a write intent from a STAGING transaction must be able to independently verify whether the transaction committed. It does this by reading the transaction record (is it STAGING?), then checking whether all referenced intents are present and durable. If they are, the transaction committed; if any is missing, the transaction may have aborted. This "intent verification" protocol means that the system's recovery path is distributed — there is no single coordinator that knows the transaction's fate. Any participant can determine it. This is both the strength (coordinator failure does not block recovery) and the complexity cost (every intent encounter may trigger a cross-range verification). The design insight is that parallel commits trades coordinator dependency for verification complexity — a trade-off that pays off enormously for throughput but requires careful engineering of the intent resolution path.

## Insight 3: The Range Is the Fundamental Atom of a NewSQL Database — Not the Row, Not the Table, Not the Node — and Every Operational Property (Replication, Latency, Throughput, Fault Tolerance) Is Determined at the Range Level

**Category:** Distributed Systems

**One-liner:** While traditional databases think in terms of rows and tables, and NoSQL systems think in terms of partitions and nodes, a NewSQL database's architecture is entirely defined by the range — a contiguous key span that is the unit of Raft consensus, replication placement, load balancing, split/merge operations, and lease management.

**Why it matters:** Understanding a NewSQL database requires understanding that every system property is a property of ranges, not of nodes or tables. Replication factor is per-range: a critical table can have 5 replicas while a log table has 3. Latency is per-range: a range whose leaseholder is in the same region as the client has sub-millisecond reads, while the same table's other ranges may have leaseholders in distant regions. Throughput is per-range: each range's Raft group can sustain roughly 2,500 writes per second, and the only way to increase write throughput is to have more ranges (via splitting). Fault tolerance is per-range: losing a node affects only the ranges that had replicas on that node, and each range independently elects a new leader. This range-centric worldview has profound implications for schema design: choosing a primary key is choosing how data maps to ranges, which determines co-location of related rows (within one range for fast transactions) vs. distribution across ranges (for parallel throughput). A sequential primary key (auto-increment) concentrates all inserts into one range, creating a hot spot. A hash-prefixed key distributes inserts but destroys range-scan locality. A composite key with a tenant prefix co-locates each tenant's data for fast single-tenant queries but may create imbalanced ranges if tenants vary greatly in size. The schema designer is, in effect, designing the range topology — and this single decision propagates to every operational characteristic of the system.

## Insight 4: MVCC Garbage Collection Is a Silent Throughput Tax — and Deferring It Trades Storage Bloat for Write Throughput Until the Collection Storm Arrives

**Category:** Storage

**One-liner:** Every MVCC write creates a new version rather than updating in place, and these old versions accumulate until garbage collection removes them — but GC competes with foreground I/O, creating a deferred cost that manifests as unpredictable latency spikes when the collection backlog becomes unsustainable.

**Why it matters:** In an LSM-tree-based NewSQL database, every UPDATE or DELETE does not remove the old value. Instead, it writes a new MVCC version at a higher timestamp. The old version persists until the GC window expires (typically 24-48 hours) and a compaction pass physically removes it. During this window, reads must skip over dead versions, range scans return and discard obsolete entries, and the LSM tree contains 2-5x the logical data size. The GC process itself is I/O-intensive: it must read SST files, identify expired versions, and rewrite cleaned files — consuming the same disk bandwidth that foreground queries need. If GC falls behind (common during sustained write bursts), the version backlog grows, read amplification increases (more dead versions to skip), and storage consumption climbs. When GC finally catches up, it creates a compaction storm — a burst of background I/O that spikes foreground latency. The production insight is that GC tuning is as critical as index design: the GC window must be long enough for point-in-time recovery needs but short enough to prevent version accumulation from degrading read performance. Teams that set a 48-hour GC window without measuring the resulting space amplification often discover that their 8 TB dataset occupies 25 TB of physical storage — and that reducing the window to 4 hours cuts storage by 60% with no functional impact.

## Insight 5: LSM Write Amplification Is the Hidden Cost of Write-Optimized Storage — and the Choice Between Tiered and Leveled Compaction Determines Whether You Pay This Cost on Writes or Reads

**Category:** Storage

**One-liner:** An LSM-tree achieves high write throughput by buffering writes in memory and flushing sorted runs to disk, but maintaining sorted order requires compaction — repeatedly rewriting data as it moves through levels — and the compaction strategy (tiered vs. leveled) determines the trade-off between write amplification, read amplification, and space amplification.

**Why it matters:** Write amplification in an LSM-tree means that a single logical write (1 KB row insert) may cause 10-30x the physical I/O due to compaction. In leveled compaction, each level is sorted and non-overlapping, so a newly flushed file must be merge-sorted into the existing level — rewriting all overlapping data. This produces low read amplification (at most one file per level to check) and low space amplification (no redundant data), but high write amplification (10-30x). In tiered compaction, files at each level are allowed to overlap, and compaction only triggers when too many files accumulate at a level. This produces low write amplification (2-4x), but higher read amplification (multiple overlapping files to check) and higher space amplification (redundant entries across files). The production implication is that the compaction strategy must match the workload: tiered compaction for write-heavy OLTP (payment processing, IoT ingestion), leveled compaction for read-heavy workloads (user-facing queries, analytics). Some production systems use a hybrid: tiered compaction for lower levels (where data churns fast) and leveled for upper levels (where data is stable). The key metric is the write amplification factor (WAF) — bytes written to disk divided by bytes written by the application. A WAF of 20x means that sustaining 50K writes/sec (each 256B) requires 256 MB/s of disk write bandwidth, which may saturate a standard NVMe drive. Understanding this relationship is essential for capacity planning.

## Insight 6: Leaseholder Placement Determines Read Latency Geography — Making the Placement Policy a Latency Routing Decision Disguised as a Replication Configuration

**Category:** Geo-Distribution

**One-liner:** Because only the leaseholder can serve strong consistent reads, the physical location of each range's leaseholder determines the read latency experienced by clients in each region — transforming what appears to be a replication configuration into a latency routing decision that directly shapes user experience.

**Why it matters:** In a multi-region NewSQL deployment, every range has replicas in multiple regions but only one leaseholder. A strong read must go to the leaseholder — there is no alternative without sacrificing consistency. If a US user queries data whose leaseholder is in Europe, they pay 80-150ms round-trip latency, even though a local replica exists 2ms away. This makes leaseholder placement the single most impactful latency decision in a multi-region deployment. The placement policy must consider access patterns: a `users` table partitioned by region should have each region's user data leased to nodes in that region. A global `config` table read by all regions might use a single leaseholder (one region fast, others slow) or follower reads (all regions fast but slightly stale). The subtlety is that writes must also go through the leaseholder, and the leaseholder must achieve Raft quorum — which may require cross-region round-trips. A leaseholder in US-East with replicas in US-West and EU-West must wait for at least one cross-region replica to acknowledge a write (~40-100ms). Pinning both leaseholder and a quorum replica to the same region (e.g., 2 replicas in US-East, 1 in US-West, 1 in EU-West with a 4-replica config) enables local quorum writes at the cost of reduced fault tolerance in the primary region. This leaseholder-as-latency-router insight is what separates teams that achieve single-digit millisecond read latency in a globally distributed database from teams that see unexplained 100ms+ latencies on simple queries.

## Insight 7: Online Schema Changes Require the Two-Version Rule that never changes — and Violating It Creates Silent Data Corruption That No Transaction Protocol Can Detect

**Category:** Distributed Transactions

**One-liner:** In a distributed database where nodes upgrade independently, at most two adjacent schema versions can coexist at any time — because if three versions are active simultaneously, a write from version V and a read from version V+2 may interpret the same row differently, corrupting data in a way that appears valid to both sides.

**Why it matters:** When adding a column with a default value, the schema change must progress through stages: (1) nodes learn the new schema but do not write the new column; (2) nodes begin writing the new column on new inserts and backfilling existing rows; (3) all nodes enforce the new column on reads and writes; (4) the old schema is dropped. At any moment during this process, two nodes may be running different stages. The two-version Rule that never changes guarantees safety: if Node A is at stage 2 (writing the new column) and Node B is at stage 1 (not yet writing it), they can coexist because stage-1 nodes simply ignore the new column in reads. But if Node A is at stage 3 (requiring the new column) and Node C is still at stage 1 (not writing it), a row written by C will lack the new column, and A will interpret this as a constraint violation or return incorrect results. The operational consequence is that schema changes must propagate in bounded time — each version fence must complete cluster-wide before the next version is introduced. A node that is too slow to upgrade blocks the entire schema change pipeline. In production, this manifests as "schema change stuck at 99%" — one slow node holding back the version fence. The mitigation is proactive monitoring of schema version distribution across nodes and automatic decommissioning of nodes that cannot keep up with version progression.

## Insight 8: Follower Reads Trade Freshness for Linear Read Scalability — but the Staleness Bound Is Not the Staleness You Will Observe

**Category:** Consistency

**One-liner:** Serving reads from follower replicas eliminates the leaseholder Slowest part of the process and scales read throughput linearly with replica count, but the "bounded staleness" guarantee (e.g., 15 seconds) is a worst-case bound — actual observed staleness depends on Raft replication lag, which is typically milliseconds under normal conditions but can spike to seconds during leader elections or network partitions.

**Why it matters:** A leaseholder serving all reads for a popular range becomes a throughput Slowest part of the process — limited by a single node's CPU, network, and I/O capacity. Follower reads allow any replica to serve reads with a bounded staleness guarantee, scaling read throughput linearly with the replication factor. If a range has 5 replicas, follower reads provide up to 5x the read throughput of leaseholder-only reads. The catch is understanding what "bounded staleness" means in practice. A follower read with a 15-second staleness bound waits until the follower's applied Raft index is within 15 seconds of the leader's current time. Under normal conditions, Raft replication lag is under 100ms, so the follower's data is nearly fresh. But during a leader election (5-10 seconds), a network partition, or a compaction storm on the follower, the applied index may fall behind. The follower read then blocks until the follower catches up — converting a "stale but fast" read into a "fresh but slow" read. Worse, if the follower is so far behind that it cannot serve within the staleness bound, the read fails. The production pattern is to monitor `raft.log.behind` per replica and route follower reads only to replicas within a healthy lag threshold, while falling back to the leaseholder for replicas that are too far behind. Teams that deploy follower reads without monitoring replication lag discover that their "always fast" reads occasionally block for seconds — exactly the behavior they were trying to avoid.

## Insight 9: Every Write Intent Imposes a Future Read Cost — Making Write Patterns a First-Class Input to Read Latency Modeling

**Category:** Performance

**One-liner:** An uncommitted write intent is not just a provisional value — it is a read-path obstacle that forces every subsequent reader to pause, look up the intent's transaction record, determine the transaction's fate, and potentially wait for or resolve the intent — adding microseconds to milliseconds of latency per encountered intent.

**Why it matters:** In MVCC-based NewSQL databases, a write does not immediately produce a committed value. It produces an intent — a provisional value tagged with a transaction ID. When a reader encounters this intent, it cannot simply skip it, because the intent's transaction might have committed (in which case the value is visible) or might still be pending (in which case the reader must wait). The reader must: (1) read the transaction record from the range that owns it (potentially a cross-range RPC), (2) determine the transaction's status (PENDING, STAGING, COMMITTED, ABORTED), and (3) either wait for the transaction to complete, resolve the intent, or skip it. In a high-contention workload where many transactions write to overlapping key ranges, readers encounter intents frequently. Each encounter adds 0.5-5ms of latency — a cross-range RPC to read the transaction record, plus potential waiting time. On a hot key with 1,000 writes/sec, every read to that key is likely to encounter at least one intent. The production consequence is that write patterns directly affect read latency: a burst of long-running transactions that write many intents can degrade read latency across the cluster even if the reads are to different keys in the same ranges. Intent resolution is eventually consistent — committed intents are asynchronously cleaned up by background resolvers — but between commit and cleanup, every reader pays the resolution cost. The mitigation is aggressive intent resolution: resolvers that run with high priority, batched resolution that cleans up all intents from a committed transaction in a single pass, and monitoring of intent count per range as a leading indicator of read latency degradation.

## Insight 10: Multi-Region Quorum Placement Creates an Inescapable Write Latency Floor — and No Optimization Can Reduce It Below the Speed of Light

**Category:** Geo-Distribution

**One-liner:** A Raft write requires majority quorum acknowledgment, and in a multi-region deployment, at least one replica must be in a remote region to survive region failure — making the minimum write latency equal to the round-trip time to the nearest remote quorum member, a physical constant that no software optimization can reduce.

**Why it matters:** Consider a 3-replica range with replicas in US-East, US-West, and EU-West. A write from US-East requires quorum (2 of 3 replicas). The leaseholder in US-East writes locally (1ms) and waits for either US-West (~40ms RTT) or EU-West (~80ms RTT). The write latency floor is 40ms — the fastest quorum is US-East + US-West. No parallel commits optimization, no pipelining, no batching can reduce this below 40ms, because the speed of light between data centers is a physical constant. A 5-replica range with 3 replicas in US-East and 1 each in US-West and EU-West achieves local quorum (3 of 5, all in US-East), reducing the write latency floor to ~2ms — but now losing the US-East region loses quorum entirely (only 2 of 5 survive), violating the multi-region survivability goal. This creates a fundamental tension: regional survivability requires cross-region quorum, which imposes a latency floor. Local quorum achieves low latency but sacrifices regional survivability. The only escape is to accept different guarantees for different data: region-pinned tables with local quorum + async replication (fast writes, non-zero RPO), and global tables with cross-region quorum (slow writes, zero RPO). The schema designer must classify every table along this spectrum, and the classification directly determines the user-perceived latency for writes to that table.

## Insight 11: Raft Group Resource Consumption Scales with Range Count, Not Data Size — Making Range Size Configuration a Cluster-Wide Resource Budget Decision

**Category:** Resource Management

**One-liner:** Each range maintains its own Raft state machine, leader election timer, heartbeat mechanism, WAL, and in-memory log buffer — and these per-range fixed costs accumulate linearly with range count, meaning a cluster with 200K ranges (small range size) consumes dramatically more memory, CPU, and goroutines than a cluster with 50K ranges (large range size) holding the same data.

**Why it matters:** The default range size (512 MB for some systems, 64 MB for others) determines how many ranges a dataset produces. A 10 TB dataset with 64 MB ranges produces ~160K ranges; with 512 MB ranges, ~20K. Each range, regardless of data size, consumes: ~2-5 MB of memory (Raft log buffer, range descriptor, lease state), one goroutine per Raft tick (heartbeat every 200ms), periodic disk I/O for Raft snapshots, and CPU for leader election timers. At 160K ranges on a 20-node cluster, each node manages ~8,000 Raft groups — consuming ~24 GB of memory just for Raft overhead, plus ~8,000 goroutines for heartbeats. The Raft heartbeat traffic alone (8,000 groups × 3 replicas × 5 heartbeats/sec) generates 120K messages per second per node. Smaller ranges improve write throughput (more parallel Raft groups, finer-grained splitting) and reduce blast radius (a failed range affects less data). But they increase memory consumption, heartbeat traffic, and GC pressure. Production systems find the sweet spot empirically: 256 MB ranges for general workloads, 64 MB for write-heavy workloads that need aggressive splitting, and 512 MB+ for read-heavy analytical workloads where write throughput per range is not a concern. The key monitoring metric is not data size or QPS but ranges-per-node — and the typical ceiling is 10,000-15,000 ranges per node before Raft overhead begins to degrade performance.

## Insight 12: Hot Ranges Cannot Be Solved by Adding Nodes — Only by Splitting the Range or Redesigning the Schema

**Category:** Scalability

**One-liner:** A hot range's write throughput is capped at ~2,500 writes/sec by its single Raft leader, and adding nodes to the cluster does nothing to increase this limit — because horizontal scaling distributes ranges across nodes but does not increase the throughput of any individual range's Raft group.

**Why it matters:** This is the most counter-intuitive limitation of NewSQL databases for teams coming from shared-nothing architectures. In a hash-partitioned system, adding nodes rebalances partitions and reduces per-node load. In a range-based NewSQL system, adding nodes allows the rebalancer to move ranges to new nodes — but a hot range stays on one node with one Raft leader. If range R1 receives 5,000 writes/sec and its Raft group can sustain 2,500, adding 10 new nodes does not help — R1 is still bottlenecked on its single leader. The solutions all involve reducing the load on that single range: (1) load-based range splitting divides R1 into R1a and R1b, each handling ~2,500 writes/sec with independent Raft groups; (2) hash-sharded indexes scatter sequential writes (auto-increment keys, timestamps) across many ranges by prepending a hash prefix; (3) application-level sharding partitions logical counters into N buckets (e.g., 16 sub-counters), each in a different range, summing on read. The production implication is that hot spot detection and remediation must be automated: the system must detect that a range's Raft leader is saturated (via proposal latency > 50ms or queue depth > 100), automatically split it at the load midpoint, and rebalance the halves to different nodes — all within seconds, not minutes. Teams that rely on manual intervention for hot spots discover that by the time an engineer diagnoses and fixes the issue, the application has already experienced minutes of elevated latency.
